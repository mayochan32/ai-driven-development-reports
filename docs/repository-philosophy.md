# リポジトリの理念

このリポジトリは、AI駆動開発に役立つ技術考察を、テーマ別の公開レポートとして積み上げるための場所である。

目的は「AI駆動開発とは何か」という総論を一度にまとめることではない。むしろ、実践や対話の中で生まれた個別の論点を、技術的な判断材料として後から参照できる形で公開していくことにある。

ここでいう技術考察とは、単なる感想やトレンド紹介ではない。開発者やチームが、AI駆動開発の設計、実装、レビュー、テスト、ドキュメント、運用を考えるときに使える判断軸、比較、注意点、実践パターンを整理したものである。

## 1. テーマ単位で考える

AI駆動開発は範囲が広い。要件定義、設計、実装、レビュー、テスト、ドキュメント、チーム運営、学習、評価、セキュリティなど、多くの論点が含まれる。

そのため、このリポジトリでは全体を一つの大きな文書にまとめない。各レポートは、ひとつのテーマ、ひとつの問い、ひとつの判断軸を中心に構成する。

各レポートは独立して読めることを重視する。一方で、複数レポートを横断して見たときに、AI駆動開発に対する考え方が徐々に立ち上がる状態を目指す。

各テーマでは、読者が次の行動や設計判断に移れることを重視する。抽象的な主張だけで終わらせず、形式の選び方、プロセスの組み方、レビュー観点、安全条件、運用上のトレードオフなど、実務に接続できる形へ落とし込む。

### テーマフォルダを配布単位にする

各テーマのフォルダは、正本MarkdownとHTMLだけでなく、そのテーマに固有の素材、評価データ、生成ツール、AIスキル、エージェント定義、依存関係を含む自己完結したパッケージとする。

利用者が一つのテーマだけを持ち出したい場合、`reports/<number>-<slug>/`を取得すれば必要な公開成果物が揃う状態を保つ。

リポジトリルートはテーマ横断の入口と運用基盤に限定する。再利用を見込んだだけの先回りした共通化は行わず、複数テーマで実利用されたものだけを共通領域へ昇格させる。

## 2. Markdownを正本にする

各レポートの正本はMarkdownで管理する。

Markdownは、人間にもAIにも読みやすく、Git差分を追いやすく、長期的な編集に向いている。出典、判断理由、仮定、未確認事項を残す形式としても扱いやすい。

正本Markdownには、次を含める。

- タイトル
- 日付
- ステータス
- タグ
- 要約
- 本文
- 出典
- 必要に応じた注意点や未確認事項

HTMLを更新する場合でも、論旨や根拠はMarkdownに戻せる状態を保つ。

## 3. HTMLを人間向けビューにする

HTMLはMarkdownの代替ではなく、人間が短時間で理解・比較・判断するためのビューである。

単なるMarkdown変換ではなく、レポートの内容に応じて、次のような表現を使う。

- 結論の初期表示
- 重要論点のカード化
- 比較表のフィルター
- タブによる切り替え
- 詳細根拠の折りたたみ
- チェックリスト
- 判断結果のJSON出力
- 図解やフロー表示

HTMLの価値は装飾ではなく、確認コストの削減にある。

## 4. 正本と表示を分離する

このリポジトリでは、次の役割分担を基本とする。

| 役割 | 主な形式 |
|---|---|
| 知識と意図の正本 | Markdown |
| 人間向けの表示・操作 | HTML |
| 判断結果や構造化データ | JSON |
| 契約や検証 | JSON Schema |
| 履歴や評価ログ | JSONL |

HTMLだけを保存すると、表示と意味が密結合し、後から再利用しにくくなる。逆にMarkdownだけでは、人間が大量の情報を短時間で比較・判断しにくい。

そのため、正本Markdownと人間向けHTMLをセットで公開する。

## 5. AIと人間の共同作業を前提にする

このリポジトリのレポートは、人間だけでなくAIにも読まれる。

Markdownは、AIが文脈や論旨を理解するための入力になる。HTMLは、人間がAIの出力や考察を確認し、判断し、次の作業へ戻すためのインターフェースになる。

この往復を前提に、レポートは「読む文書」であると同時に「次の作業を生む資料」として設計する。

## 6. 安全性と追跡可能性を軽視しない

AI生成HTMLは、実行可能な成果物である。便利な一方で、外部通信、スクリプト、埋め込み、機密情報の混入に注意が必要である。

HTMLを作るときは、次を原則とする。

- 外部依存を避ける
- 外部通信をしない
- 秘密情報を埋め込まない
- 可能な範囲でCSPを設定する
- 元Markdownや元データへ戻れるようにする
- 出典、仮定、未確認事項を表示から隠さない

美しい表示は、正しさの証拠ではない。見やすさと検証可能性を両立させる。

## 7. 完成より更新可能性を重視する

AI駆動開発の状況は変化が速い。したがって、このリポジトリのレポートは、最初から完全な最終版である必要はない。

重要なのは、後から読み返し、修正し、HTMLビューを再設計し、出典を追加できる状態にしておくことである。

各レポートにはステータスを持たせる。

- `draft`: 初稿または検討中
- `review`: 公開前レビュー中
- `published`: 公開済み
- `archived`: 古くなったため参照用

## 8. このリポジトリで大事にする判断基準

- その文書は、次に誰が読むのか
- その成果物は、どれくらいの期間使われるのか
- それは正本なのか、派生ビューなのか
- 人間の判断コストは下がっているか
- AIが次の作業に使える構造になっているか
- 出典、仮定、未確認事項が追えるか
- Gitで変更の理由を追えるか

この判断基準に沿って、テーマ別レポートを継続的に追加していく。
