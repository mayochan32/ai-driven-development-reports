# AGENTS.md

このリポジトリで作業するAIエージェント向けの運用ルールです。

## リポジトリの目的

AI駆動開発に役立つ技術考察を、テーマ別レポートとして公開する。総論を一つにまとめるのではなく、独立した論点ごとに、正本Markdownと人間向けHTMLをセットで管理する。

各レポートは、開発者やチームがAI駆動開発の設計、実装、レビュー、テスト、ドキュメント、運用を考えるときの判断材料になることを目指す。

## 最重要方針

- 各レポートは必ず `<number>-<slug>.md` と `<number>-<slug>.html` のセットで作る。
- Markdownファイルを正本とする。HTMLは正本ではなく、人間が理解・比較・判断しやすくする派生ビューとする。
- ファイル単体でダウンロードしても内容が分かるように、MarkdownとHTMLにはテーマに沿った固有名を付ける。`report.md` や `index.html` のような汎用名は使わない。
- HTMLはMarkdownの単純変換にしない。本文の主張を、人間の確認コストが下がる画面として再設計する。
- 出典、仮定、未確認事項、生成日時、安全上の注意を隠さない。
- レポートはテーマ単位で独立させ、全体論や大きなREADMEに吸収しない。
- `reports/<number>-<slug>/` を、そのテーマの配布、保守、再現、AI連携の単位とする。
- テーマ固有の素材、ツール、依存定義、スキル、エージェント設定をリポジトリルートへ置かない。
- あるファイルを複数テーマで実際に利用するまでは、共通化を先回りしない。

## 標準配置

```text
reports/<number>-<slug>/
├── README.md
├── <number>-<slug>.md
├── <number>-<slug>.html
├── manifest.json
├── assets/
├── tools/
├── skills/
├── .claude/
├── .github/
├── AGENTS.md
├── package.json
└── package-lock.json
```

`README.md`、正本Markdown、人間向けHTML、`manifest.json` は必須とする。その他のディレクトリとファイルは、そのテーマで必要な場合だけ作成する。

テーマフォルダだけをダウンロードした状態でも、公開成果物の確認と、明記された再生成・検証手順を実行できる構成にする。テーマ固有コマンドはテーマフォルダを作業ルートとして記述する。

リポジトリルートには、全体の入口、カタログ、共通方針、共通スキーマ、テンプレートだけを置く。共通領域へ移すのは、2つ以上のテーマが同じ実装を利用し、テーマ間で共有する方が保守しやすいと確認できた場合に限る。

## 作業手順

1. 素材文書を読む。
2. テーマ、主張、対象読者、公開時の注意点を確認する。
3. `reports/<number>-<slug>/<number>-<slug>.md` に正本Markdownを作る。
4. 同じディレクトリに `<number>-<slug>.html` を作る。
5. テーマの入口となる `README.md` と `manifest.json` を作る。
6. 素材、ツール、スキル、エージェント、依存定義が必要なら、すべて同じテーマフォルダ内へ置く。
7. HTMLは次の観点で設計する。
   - 結論が初期表示で分かる
   - 比較表は視覚的に探索できる
   - 詳細や根拠は折りたためる
   - 必要ならフィルター、タブ、チェックリスト、JSON出力を用意する
   - 外部依存は避ける
   - スクリプトを入れる場合は用途を限定し、外部通信をしない
8. ルートの `README.md` と `catalog.json` の一覧を更新する。
9. `node scripts/validate-theme-packages.mjs` で全テーマの最低構成とカタログ参照を確認する。
10. テーマフォルダを作業ルートとして、構文、リンク、DOM参照、ファイル配置、再生成手順を確認する。

## HTML作成時の注意

- HTMLビューは、配布しやすさを優先して1ファイル完結を基本とする。
- CSSとJavaScriptは原則としてHTMLファイル内に埋め込み、別ファイルに分けない。
- HTMLファイル単体をコピーしても表示・操作できる状態を保つ。
- 外部CDN、外部画像、外部フォントに依存しない。
- `connect-src 'none'` を含むCSPを可能な範囲で設定する。
- UIは装飾より判断支援を優先する。
- 色だけで意味を伝えない。ラベルや説明を併用する。
- 美しい表示を正しさの証拠にしない。元データ、出典、仮定へ戻れるようにする。

## 変更時の姿勢

- 既存レポートの正本性を壊さない。
- ユーザーが渡した考察の論旨を尊重し、勝手に総論化しない。
- 大きな構成変更をする場合は、どの公開URLや参照が変わるかを明示する。
