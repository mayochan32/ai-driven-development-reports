Kiro、cc-sdd、Spec Kitをどう使い分けるか

3つのSpec-Driven Developmentフレームワークを、公開情報による机上調査、同一要件によるエミュレーション実証、開発担当者視点、選定判断の観点から比較します。

要点

3者はいずれもSDDを支援しますが、同じ方向を向いているわけではありません。

Kiroは製品統合、cc-sddは実装規律、Spec Kitは組織標準。この3軸で見ると違いがつかみやすい。

Kiro

AWSが提供する統合的なエージェンティック開発環境。

製品統合SpecsHooks

cc-sdd

Kiro風SDDを複数エージェントで使うOSSハーネス。

TDD独立レビューBoundary-first

Spec Kit

組織標準を埋め込むGitHub公式SDDツールキット。

constitution30+ agentsIssues連携

読むときの注意

  • 実証は製品実機ではなく、同一要件によるエミュレーションです。
  • KiroのAgent HooksやIDE統合の実動作は対象外です。
  • 導入判断では自組織の実案件によるPoCが必要です。

用途別推奨

重視するものを選ぶと、相性のよい候補と理由を表示します。

推奨 Kiro

3者の性格

Kiro

Specs、Hooks、課金、管理、AWS移行導線まで含む製品型。チームで環境を揃えやすい組織に向く。

  • EARS形式の要件
  • requirements / design / tasks
  • Quick Planで軽量化可能
  • クレジット課金

cc-sdd

タスク毎TDD、独立レビュー、自動デバッグを組み込む実装規律型。欠陥コストが高い開発に向く。

  • 8エージェント対応
  • 17 skills
  • Boundary-first
  • Kiro specs互換

Spec Kit

constitution、presets、extensionsで組織標準をプロセス化する標準化型。GitHub基盤との接続に向く。

  • 30+ integrations
  • clarify / analyze / checklist
  • taskstoissues
  • uv / Python CLI

限界

実機検証ではない

KiroのAgent Hooks、IDE統合、承認UI、実サブエージェント並列実行は対象外。結果はエミュレーション条件下の比較として読む。

開発担当者視点

同じ機能を作るとき、開発者の日常作業はかなり変わります。

Kiro

開発者の感覚は「レビュアー兼承認者」。生成されたrequirements/design/tasksを読み、仮定と受入基準を検査する。

読む量が多い承認品質が重要IDE内完結

cc-sdd

開発者の感覚は「仕様の共同執筆者＋最終検収者」。実装中はTDD、独立レビュー、自動デバッグが回る。

上流合意実装は自律境界注釈

Spec Kit

開発者の感覚は「仕様アーキテクト」。constitution、clarify、analyzeで上流文書の質を高める。

上流が厚いGitHub連携CLI学習

認知負荷の中心仮定の検査上流の仕様合意上流文書の執筆・整合性維持

向いている開発

Kiroが向く開発
  • AWS基盤やAWS契約との親和性を重視する開発
  • 製品サポート、チーム管理、課金管理を重視する組織
  • PdM、QA、開発がSpecsを共有成果物として扱う職能横断チーム
cc-sddが向く開発
  • 品質規律を重視する実装フェーズの自動化
  • テスト必須のバックエンドや欠陥コストが高いシステム
  • すでにClaude Code、Codexなどのエージェント環境を使っているチーム
Spec Kitが向く開発
  • 組織標準や規制要件をプロセスとして強制したい開発
  • GitHub Enterprise、GitHub Issues、Azure DevOpsなどとの接続を重視する組織
  • エージェントを固定したくない、または複数エージェントが混在する組織

検証項目見るべき指標

監査性要件からタスク、コード、テストへの追跡可能性
