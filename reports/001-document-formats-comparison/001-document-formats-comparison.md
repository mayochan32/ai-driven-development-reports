# AI駆動開発における文書・データ・表示形式の比較

## このレポートの位置づけ

このレポートは、AI駆動開発における成果物の形式選定を扱う。対象は、AIエージェントへの入力、AIからの出力、レビュー結果、設計資料、構造化データ、可視化ページ、意思決定ログなどである。

ここでは「MarkdownかHTMLか」という二択ではなく、**正本、構造化データ、契約、表示、操作、記録を分けて設計する**という観点から整理する。

なお、各形式の評価は特定のプロジェクトやツールに依存しうる。したがって本稿の比較表は絶対的な序列ではなく、AI駆動開発で形式を選ぶときの判断補助として扱う。

## 1. エグゼクティブサマリー

AI駆動開発では、資料の形式を「どれが最も優れているか」という単一の尺度で選ぶべきではない。重要なのは、次の3点である。

1. **次に誰が読むのか**
   - AI、RAG、CI、Gitなどの機械が読む
   - 人間が確認・比較・判断する
2. **その成果物をどの程度の期間保持するのか**
   - 数時間だけ使う一時成果物
   - 数週間から数か月使う作業資料
   - 半年以上維持する正本・監査対象
3. **その形式が何の役割を担うのか**
   - 意図・背景を記録する知識層
   - 型付きデータを扱う意味・契約層
   - 人間が理解・操作する表示層

従来は、MarkdownをAI駆動開発の中心形式とする考え方が強かった。これは現在も正しい。しかし、AIが大量の分析結果やレビューを短時間で生成するようになると、ボトルネックは「資料の作成」から「人間による確認と判断」へ移る。

長いMarkdownレポートは、機械には読みやすくても、人間には必ずしも消費しやすくない。HTMLは、色、空間配置、タブ、フィルター、SVG、フォーム、スライダーなどを用いて、AIの出力を人間が短時間で確認できる形へ変換できる。さらに、資料そのものを一時的な操作ツールにし、人間の判断結果をJSONやプロンプトとしてAIへ戻すこともできる。[^1][^2]

したがって本レポートでは、次の役割分担を基本方針とする。

> **Markdownは知識と意図の正本、JSONは処理可能な意味構造、JSON Schemaは契約、HTMLは人間向けの可視化・操作・意思決定インターフェース。**

HTMLはMarkdownの後継や全面的な代替ではない。両者は「次の読者」と「成果物の役割」が異なる並列の選択肢である。

## 2. 形式選定の新しい考え方

### 2.1 「何を書くか」だけでなく「次に誰が読むか」で選ぶ

| 次の読者 | 優先すべき特性 | 主な形式 |
|---|---|---|
| AI・LLM | トークン効率、明示的な見出し、ノイズの少なさ | Markdown、JSON、JSONL |
| RAG・検索基盤 | 分割しやすさ、安定した見出し、メタデータ | Markdown＋Front Matter、JSONL |
| Gitレビュー担当者 | 行単位差分、手修正のしやすさ | Markdown、TOML、ソースコード |
| CI・プログラム | 厳密な構文、型、検証可能性 | JSON、JSON Schema、YAML、TOML |
| 人間の意思決定者 | 視認性、比較、要約、段階的開示 | HTML、ダッシュボード |
| 人間の編集者 | 直接編集、コメント、変更履歴 | Markdown、AsciiDoc |
| 人間とAIの共同作業 | 操作結果を構造化して返せること | HTML＋JSON |

### 2.2 HTMLの価値は「装飾」ではなく「確認コストの削減」

HTMLの価値を、文字色やカード表示だけと捉えるのは不十分である。

HTMLが提供できるのは、次のような**意思決定支援機能**である。

- 重大項目だけを初期表示する
- 詳細な根拠を折り畳む
- 複数案を横並びやタブで比較する
- コスト、速度、保守性などの軸でフィルターする
- タイムラインや影響範囲を視覚化する
- パラメータを変更して結果を試算する
- 承認、却下、保留、タグ付けを行う
- 操作結果をJSONやAI向けプロンプトとして出力する

つまりHTMLは、単なる「読みやすい文書」ではなく、場合によっては**AIと人間の間に置く一時的な専用アプリケーション**になる。[^2]

### 2.3 HTMLはAIの分析能力そのものを高めるわけではない

HTML化によって主に改善されるのは、次の領域である。

- 情報配置
- 視認性
- 比較のしやすさ
- 人間の理解速度
- 人間からAIへのフィードバック入力

一方で、次のものはHTML化だけでは改善しない。

- 元情報の正確性
- AIの推論の妥当性
- データの完全性
- 結論の公平性
- 根拠の信頼性

むしろ、美しく整えられた誤情報は、粗いテキストより信頼できるように見える危険がある。そのため、HTMLレポートでは出典、生成日時、仮定、未確認事項、データ欠損、推測と事実の区別を明示する必要がある。

## 4. MarkdownとHTMLの再評価

### 4.1 基本比較

| 観点 | Markdown | HTML |
|---|---|---|
| 主な読者 | AI、人間の編集者、Git | 人間の確認者、意思決定者 |
| 得意な役割 | 意図、要件、設計理由、記録 | 可視化、比較、操作、共有 |
| 記述量 | 少ない | 多い |
| トークン効率 | 高い | 低い |
| 人間による直接編集 | 容易 | 相対的に難しい |
| Git差分 | 読みやすい | 属性や空白で荒れやすい |
| 視覚表現 | 基本的 | 非常に高い |
| インタラクション | ほぼ不可 | 可能 |
| フォーム・入力UI | 不可 | 可能 |
| 色・空間配置 | 限定的 | 自由度が高い |
| アクセシビリティ | 限定的 | セマンティックHTMLとARIAで対応可能 |
| スクリプト実行 | 不可 | 可能だがセキュリティリスクあり |
| 長期保存の正本 | 適する | 条件付き |
| 一時的な専用ツール | 不向き | 非常に適する |

### 4.2 「MarkdownからHTMLへ変換」と「HTMLを直接設計」の違い

単純なMarkdownからHTMLへの変換では、通常は次の構造しか得られない。

- 見出し
- 段落
- 箇条書き
- 表
- コードブロック

一方、最初から人間の意思決定用HTMLとして設計すると、次のような表現を選べる。

- タブ式比較ボード
- フィルター可能な一覧
- 根拠を段階的に開示するアコーディオン
- パラメータシミュレーター
- SVGによる自由な図解
- 承認・却下・優先順位付けUI
- 編集結果のJSON出力

したがって、「MarkdownをHTMLに変換すれば同じ」というわけではない。HTMLの表現能力を活用するには、AIへ最初から**誰が何を判断するための画面なのか**を伝える必要がある。[^1]

ただし、HTMLを直接生成することが常に正解というわけでもない。重要な成果物では、背後に構造化された正本を持つ方が安全である。

## 5. 推奨する責任分離アーキテクチャ

ここまでの整理を実際の開発プロジェクトに適用する場合、重要なのは「3つのファイル形式に分ける」ことではない。知識、意味、契約、表示、フィードバック、記録という責任を分け、それぞれに適した形式を選ぶことである。

### 5.1 基本構造

```text
知識層：知識・意図の正本
Markdown / AsciiDoc
        ↓
意味層：処理可能な意味構造
JSON / JSONL / OpenAPI
        ↓
契約層：型・制約・API契約
JSON Schema / OpenAPI
        ↓
表示層：人間向けの表示・操作
HTML / CSS / SVG / JavaScript
        ↓
フィードバック層：人間の判断結果
decision.json / feedback.md
        ↓
記録層：実行履歴・監査証跡
JSONL / Markdown
```

### 5.2 各層の責任

| 層 | 主な形式 | 責任 |
|---|---|---|
| 知識層 | Markdown、AsciiDoc | 背景、目的、判断理由、要件 |
| 意味層 | JSON、JSONL | 機械が扱う項目、状態、関係 |
| 契約層 | JSON Schema、OpenAPI | 型、必須項目、許容値、制約 |
| 表示層 | HTML、CSS、SVG | 人間の理解、比較、操作 |
| フィードバック層 | JSON、Markdown | 人間の判断をAIへ返す |
| 記録層 | JSONL、Markdown | 実行履歴、監査証跡、意思決定ログ |

### 5.3 例：AIコードレビュー

```text
review-result.json
├── summary
├── findings[]
│   ├── severity
│   ├── file
│   ├── line
│   ├── category
│   ├── evidence
│   └── recommendation
└── test_status
```

このJSONを正本として、次を派生生成する。

```text
review-result.json
├── review.html
├── review.md
├── GitHubコメント
├── Slack通知
└── metrics.jsonl
```

HTML画面では、重大度フィルター、ファイル別表示、根拠の展開、承認・却下を提供できる。一方、分析結果そのものはJSONとして保存されるため、HTMLのデザインを変更しても意味データは失われない。

## 6. 直接HTML生成と構造化データ経由の使い分け

### 6.1 AIに直接HTMLを書かせてよいケース

| ケース | 適性 | 理由 |
|---|---:|---|
| 一回限りの比較ページ | ◎ | 短期間で選択肢を比較できる |
| UIプロトタイプ | ◎ | 見た目と操作をその場で確認できる |
| データ探索 | ◎ | フィルターやグラフを自由に構成できる |
| パラメータ試算 | ◎ | スライダーや入力欄を利用できる |
| 学習用シミュレーター | ◎ | 操作による理解を促進できる |
| 一時的な編集ツール | ◎ | 操作結果をJSONとして書き出せる |
| 会議用の分析ページ | ○ | その場の意思決定を支援できる |
| PR説明ページ | ○ | 重要差分や影響範囲を視覚化できる |

### 6.2 HTMLを唯一の正本にしない方がよいケース

| ケース | 推奨する正本 |
|---|---|
| 製品要件 | Markdown |
| ADR | Markdown |
| API定義 | OpenAPI |
| データ契約 | JSON Schema |
| アプリ設定 | TOML / YAML |
| テストケース | コード / JSONL |
| RAG投入資料 | Markdown＋メタデータ |
| 監査証跡 | JSONL＋不変ログ |
| 長期製品ドキュメント | Markdown / AsciiDoc |
| CIログ | テキスト / JSONL |

## 7. 成果物の寿命による選択

| 寿命 | 典型例 | 推奨形式 |
|---|---|---|
| 数分〜数時間 | 比較画面、試算ツール、探索UI | 単一HTML |
| 数日 | レビュー、会議用ダッシュボード | HTML＋JSON |
| 数週間〜数か月 | 仕様案、提案書、分析レポート | Markdown＋HTMLビュー |
| 半年以上 | 製品仕様、ADR、運用手順 | Markdown / AsciiDoc |
| 長期監査 | 実行履歴、判断履歴、評価データ | JSONL＋スキーマ |

成果物の寿命が長くなるほど、直接編集、差分管理、再生成可能性、監査性が重要になる。そのため、長期成果物ではMarkdownや構造化データを正本とし、HTMLは派生表示とする方が堅い。

## 8. 各形式の向き不向き

### 8.1 Markdown

#### 向いている用途

- README
- 要件定義
- ADR
- 設計思想
- AIエージェントへの指示
- プロンプト
- 議事録
- RAG投入資料
- 人間とAIの共同編集

#### メリット

- 人間とAIの双方が読みやすい
- トークン効率が高い
- Git差分が明確
- 手修正が容易
- コード、表、画像、Mermaidを埋め込める

#### 不向き

- 厳密な型保証
- 高度なレイアウト
- インタラクティブ操作
- 大量の独立レコード
- 複雑なUI

---

### 8.2 HTML

#### 向いている用途

- 人間向け分析レポート
- 比較検討ページ
- ダッシュボード
- UIプロトタイプ
- コードレビュー補助
- 学習コンテンツ
- 一時的な編集ツール
- Human-in-the-loopの操作面

#### メリット

- 情報を空間的に配置できる
- 色や強調を利用できる
- タブ、フィルター、スライダー、フォームを使える
- SVGによる自由な図解が可能
- 人間の判断結果を構造化して取得できる
- ブラウザで即時確認できる

#### 不向き・注意点

- トークン消費が大きい
- Git差分が読みにくい
- 人間が直接直しにくい
- デザイン品質が指示とテンプレートに依存する
- 誤った情報をもっともらしく見せる危険がある
- スクリプトや外部通信によるセキュリティリスクがある

---

### 8.3 JSON

#### 向いている用途

- AIの構造化出力
- API入出力
- エージェント間メッセージ
- レビュー結果
- テスト結果
- HTML表示の元データ

#### メリット

- キーと値が明示される
- 配列、数値、真偽値を区別できる
- 多くの言語で処理できる
- HTML、Markdown、通知などへ派生しやすい

#### 不向き

- 長文の背景説明
- 人間向けの正式文書
- コメントを多く含む設定

---

### 8.4 JSON Schema

#### 向いている用途

- AI出力の検証
- APIデータの制約
- 設定ファイルの検証
- エージェント間契約

#### メリット

- 必須項目と型を定義できる
- 列挙値、範囲、形式を制限できる
- AI出力の揺れを自動検出できる

#### 不向き

- 背景や意図の説明
- 人間向けの読み物

---

### 8.5 YAML

#### 向いている用途

- CI/CD
- 複雑な設定
- ワークフロー
- エージェント定義
- インフラ構成

#### メリット

- JSONより人間が読みやすい
- コメントを書ける
- 階層や配列を簡潔に表現できる

#### 不向き・注意点

- インデントミス
- 暗黙の型変換
- 巨大なYAML
- 厳密さを最優先する設定

---

### 8.6 TOML

#### 向いている用途

- アプリ設定
- モデル設定
- パス
- 機能フラグ
- ビルド設定

#### メリット

- 型が比較的明確
- インデントに意味がない
- Git差分が読みやすい
- YAMLより事故が少ない

#### 不向き

- 複雑なワークフロー
- 深い階層
- 長文プロンプト

---

### 8.7 JSONL

#### 向いている用途

- 会話履歴
- AI実行ログ
- 評価データ
- 学習データ
- イベントストリーム
- 監査記録

#### メリット

- 1行1レコード
- 逐次処理しやすい
- 追記しやすい
- 大量データに向く
- Gitで変更行を追いやすい

#### 不向き

- 長い説明文書
- 全体構造を一目で確認する用途

---

### 8.8 CSV / TSV

#### 向いている用途

- 一覧
- 用語集
- 評価結果
- 対応表
- 集計データ

#### メリット

- 非常にコンパクト
- Excelや分析ツールで扱いやすい
- 大量の平坦データに強い

#### 不向き

- 階層構造
- 配列
- コード
- 長文やセル内改行が多いデータ

---

### 8.9 AsciiDoc

#### 向いている用途

- 大規模技術文書
- 製品マニュアル
- 書籍
- 複数ファイルの再利用
- PDFやEPUBへの出力

#### メリット

- 相互参照
- 条件付きコンテンツ
- 高機能な表
- 文書の部品化

#### 不向き

- 小規模README
- 導入コストを抑えたいチーム
- 軽量なAI指示

---

### 8.10 OpenAPI

#### 向いている用途

- REST API仕様
- SDK生成
- モック生成
- APIテスト
- AIツール定義

#### メリット

- 入出力と認証を機械可読に定義できる
- API実装と文書を同期しやすい
- 検証や生成に利用できる

#### 不向き

- 一般的な要件書
- UI設計
- 議事録

---

### 8.11 Mermaid

#### 向いている用途

- シーケンス図
- 構成図
- 状態遷移図
- ER図
- 処理フロー

#### メリット

- テキストとして管理できる
- AIが構造を修正しやすい
- Git差分を確認できる
- Markdownへ埋め込める

#### 不向き

- 自由曲線や複雑な重なり
- 精密な空間配置
- 高度なインタラクティブ図

---

### 8.12 Jupyter Notebook

#### 向いている用途

- データ分析
- モデル比較
- プロンプト評価
- RAG評価
- グラフ
- 実験記録

#### メリット

- 説明、コード、結果を一体化できる
- 人間が結果を確認しやすい
- AIと反復実験しやすい

#### 不向き・注意点

- 実行順序に依存する
- 隠れた状態が残る
- Git差分が読みにくい
- 本番コードの正本には向かない

## 9. HTMLを安全に運用するための条件

AI生成HTMLは実行可能な成果物であるため、Markdownより強い安全対策が必要になる。

特に注意すべき点は、HTMLが「読むだけの文書」に見えても、JavaScript、外部リソース、フォーム、iframe、ストレージ、Cookie、ブラウザ拡張などと結びつくと、実行環境の一部になることである。AI生成HTMLをそのまま社内システムと同じOriginで開くと、意図しない外部通信や認証情報への接触を見落とす可能性がある。

### 9.1 最低限の対策

- 社内システムと異なるOriginで表示する
- `iframe sandbox`で隔離する
- Content Security Policyを設定する
- 外部通信先を許可リスト方式にする
- `<script>`、`fetch`、外部画像、iframeを検査する
- Cookieや機密情報へアクセスできない環境で開く
- CDN依存を避けるか、利用先を固定する
- 秘密情報をHTMLへ埋め込まない
- HTMLの元データと生成条件を保持する

`iframe sandbox`は、信頼できないHTMLを隔離表示するための基本的な手段である。スクリプト実行が必要な場合でも、安易に`allow-same-origin`を付けると隔離の意味が弱くなるため、用途ごとに許可する能力を絞る。CSPは外部スクリプト、画像、通信先、フレーム埋め込みなどを制限する防御層として使う。[^3][^4][^5]

### 9.2 推奨構成

```text
AI生成データ
    ↓
JSON Schemaで検証
    ↓
HTML生成
    ↓
静的解析
├── 外部URL検査
├── script検査
├── fetch検査
└── iframe検査
    ↓
隔離プレビュー
├── 別Origin
├── CSP
├── sandbox
└── 社内Cookieなし
```

### 9.3 保存すべき生成情報

HTMLを派生成果物として扱う場合、少なくとも次を残す。

- 元データ
- 生成に使ったプロンプトまたは指示
- 生成日時
- 生成モデルまたはツール
- 外部通信の有無
- 手修正の有無
- 正本への戻し方

これにより、HTMLを見て判断した後でも、どの情報からその画面が作られたのかを追跡しやすくなる。

## 10. HTML資料の品質を安定させる設計条件

単に「見やすいHTMLにして」と指示すると、過剰なカード、グラデーション、アニメーション、低い情報密度などが発生しやすい。

HTML生成時には、次を明示する。

```text
目的：
5分以内にGo / No-Go判断を行う

対象読者：
開発責任者とプロダクトオーナー

初期表示：
結論、重大リスク、推奨アクション

詳細：
根拠とログは折り畳む

比較軸：
コスト、速度、保守性、拡張性

色：
状態と重大度の表現だけに使用する

操作：
フィルター、根拠展開、承認、却下

出力：
判断結果をdecision.jsonとして保存する

制約：
外部通信なし、単一HTML、印刷可能、キーボード操作対応
```

重要なのはHTMLという形式ではなく、**誰が、何を、どの時間内で判断するか**を設計することである。

## 13. 避けるべきアンチパターン

### 13.1 すべてをMarkdownで表現する

問題：

- 大量レポートが読まれない
- 比較や探索に時間がかかる
- 人間の判断が非構造化テキストになる

改善：

- 重要なレビューや比較はHTMLビューを生成する
- 判断結果はJSONとして取得する

### 13.2 すべてをHTMLにする

問題：

- Git差分が悪化する
- 原本が編集しにくい
- AI入力時のトークンが増える
- 長期保守が難しい

改善：

- MarkdownまたはJSONを正本にする
- HTMLは派生表示とする

### 13.3 HTMLだけを保存する

問題：

- 表示と意味データが密結合する
- デザイン変更時に再利用しにくい
- 自動処理や監査が難しい

改善：

- 元データをJSONで保存する
- HTMLを再生成可能にする

### 13.4 色だけで意味を示す

問題：

- AI、スクリーンリーダー、白黒印刷で意味が失われる

改善：

- `Critical`、`Warning`、`Approved`などの文字ラベルを併用する

### 13.5 美しい表示を正しさの証拠とみなす

問題：

- 誤った分析がもっともらしく見える
- 出典や仮定が隠れる

改善：

- 出典、仮定、確信度、未確認事項を表示する
- 元データへ遡れるようにする

## 14. 最終提言

AI駆動開発の資料形式は、次の原則で選ぶとよい。重要なのは、形式を好みで選ぶことではなく、**次の読者、正本性、検証可能性、判断コスト**を分けて考えることである。

### 原則1：次の読者で決める

- AI、RAG、Gitが読む：Markdown
- プログラムが処理する：JSON
- 制約を検証する：JSON Schema
- 人間が確認・比較・判断する：HTML

### 原則2：正本と表示を分離する

- 正本：Markdown、JSON、OpenAPI
- 表示：HTML
- 判断結果：JSONまたはMarkdown
- 履歴：JSONL

### 原則3：成果物の寿命で決める

- 短命：直接HTML
- 中期：Markdown / JSON＋HTML
- 長期：Markdown / AsciiDoc / スキーマ
- 監査：JSONL＋不変ログ

### 原則4：HTMLを「軽い専用ツール」として使う

HTMLの本当の価値は、文書の装飾ではない。AIの大量出力を人間が確認し、比較し、操作し、判断結果をAIへ返すためのHuman-in-the-loopインターフェースにある。

### 結論

> **説明はMarkdown、設定はTOMLまたはYAML、データはJSON、契約はJSON Schema、履歴はJSONL、図はMermaid、人間の確認と操作はHTML。**

この構成により、次の3つを同時に満たせる。

- AIが理解しやすい
- プログラムが検証・処理しやすい
- 人間が短時間で判断しやすい

MarkdownとHTMLの関係は競合ではない。

> **Markdownは知識を残すための形式であり、HTMLは知識を人間が利用するためのインターフェースである。**
