要点

このレポートの結論は、形式の優劣を一つに決めることではなく、成果物の役割を分けることです。

Markdownは知識と意図の正本、JSONは処理可能な意味構造、JSON Schemaは契約、HTMLは人間向けの可視化・操作・意思決定インターフェース。
次の読者 AI、RAG、Git、人間の判断者で求める形式は変わる。
正本と表示 長期保存する正本と、人間が確認する表示を分離する。
成果物の寿命 短命ならHTML、長期ならMarkdownやスキーマを重視する。
判断コスト HTMLは装飾ではなく、確認・比較・操作のために使う。

標準構成

説明はMarkdown、設定はTOMLまたはYAML、データはJSON、契約はJSON Schema、履歴はJSONL、図はMermaid、人間の確認と操作はHTML。

正本: Markdown データ: JSON 契約: Schema 表示: HTML 履歴: JSONL

判断理由

    形式比較

    形式ごとの得意領域を、役割で絞り込めます。評価は本文の比較表を人間が読みやすい単位へ分解したものです。

    責任分離アーキテクチャ

    AI駆動開発では、知識、意味、契約、表示、フィードバック、記録を分けると再利用と検証がしやすくなります。

    知識層

    背景、目的、判断理由、要件を残す。

    Markdown / AsciiDoc

    意味層

    項目、状態、関係、評価結果を機械可読にする。

    JSON / JSONL

    契約層

    型、必須項目、許容値、API境界を検証可能にする。

    JSON Schema / OpenAPI

    表示層

    人間が短時間で確認し、比較し、操作する。

    HTML / CSS / SVG / JavaScript

    フィードバック層

    承認、却下、保留、タグ、優先度を構造化する。

    decision.json / feedback.md

    記録層

    実行履歴、判断履歴、監査証跡を追記可能に残す。

    JSONL / Markdown

    HTML安全条件

    AI生成HTMLは実行可能な成果物です。便利さだけでなく、隔離、外部通信、元データの追跡可能性を確認します。

    未確認の安全条件があります。

    避けるべきアンチパターン

    形式の選定を誤ると、読まれない資料、編集できない正本、もっともらしい誤情報が生まれます。

    すべてをMarkdownで表現する

    大量レポートが読まれず、比較や探索に時間がかかり、人間の判断が非構造化テキストに残る。重要なレビューや比較はHTMLビューにし、判断結果はJSONとして取得する。

    すべてをHTMLにする

    Git差分が荒れ、原本が編集しにくくなり、長期保守が難しくなる。MarkdownまたはJSONを正本にし、HTMLは派生表示として扱う。

    HTMLだけを保存する

    表示と意味データが密結合し、デザイン変更や監査に弱くなる。元データ、生成条件、判断結果を別に保存する。

    美しい表示を正しさの証拠とみなす

    誤った分析が信頼できるように見える。出典、仮定、確信度、未確認事項、元データへの導線を画面上に残す。

    判断JSON

    上の選択と安全チェックの状態を、AIへ戻せる構造化データとして出力します。
