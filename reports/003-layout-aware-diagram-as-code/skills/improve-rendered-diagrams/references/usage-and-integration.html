導入方法を選ぶ

図を扱う頻度と、既存エージェントの責務に合わせて選択する。

LOW FREQUENCY

必要なときだけ呼ぶ

既存設定は変えず、プロンプトでスキルまたは専門エージェントを指定する。

STANDARD

既存エージェントへ組み込む

設計書や文書作成の完了条件として、レンダリングと視覚評価を追加する。

COMPLEX DIAGRAMS

専門エージェントへ委譲する

候補生成と評価ログを別コンテキストへ分離し、既存エージェントは意味と採否を管理する。

テーマ単独で利用可能: このガイドのパスとコマンドは、003-layout-aware-diagram-as-code/を作業ルートとする。スキル、エージェント定義、レンダラー、評価基準はこのフォルダ内に同梱されている。
重要: 構文検査やレンダリング成功だけでは視覚品質を確認できない。PNGを画像として開けない場合は「視覚評価未完了」とする。

環境別の使い方

プロジェクトへ共有する場合と、個人環境で再利用する場合で配置先が異なる。

用途配置・指定
個人スキル~/.codex/skills/improve-rendered-diagrams/
呼び出し$improve-rendered-diagrams をプロンプトで指定
常時ルールAGENTS.md に図変更時の必須工程を記載
$improve-rendered-diagrams を使って docs/order-flow.mmd を評価・改善してください。
ノード、分岐条件、並列性、失敗経路は変更しないでください。
用途配置・指定
同梱構成skills/improve-rendered-diagrams/ と .claude/agents/improve-rendered-diagrams.md
自動プリロード共通スキルを .claude/skills/improve-rendered-diagrams/ へコピー
個人利用~/.claude/skills/ と ~/.claude/agents/
既存エージェントfrontmatterの skills へスキル名を追加
claude --agent improve-rendered-diagrams
skills:
  - improve-rendered-diagrams
用途配置・指定
同梱構成skills/improve-rendered-diagrams/ と .github/agents/improve-rendered-diagrams.agent.md
プロジェクトスキル共通スキルを .github/skills/improve-rendered-diagrams/ へコピー
CLI個人利用~/.copilot/skills/ と ~/.copilot/agents/
既存エージェント本文から共通スキルと完了条件を参照
copilot --agent improve-rendered-diagrams

レンダラーを確認する

node skills/improve-rendered-diagrams/scripts/render-diagram.mjs --check

不足するツールは無断でインストールせず、人間の承認を得る。

依頼文の例

新しい図を作る

EC注文処理のフローチャートをMermaidで作成してください。
在庫確保と不正検知は並列処理です。どちらかが失敗したら注文を中止します。
improve-rendered-diagramsを使い、レンダリング画像を評価してから最良版を提示してください。

既存の図を改善する

docs/order-flow.mmdを改善してください。
ノード、分岐条件、並列性、失敗経路は変更しないでください。
画像を実際に確認し、対称性を含む27点評価と改善前後の成果物を残してください。

同じ評価基準で比較してください。必須違反がなく、最も誤読しにくい形式を採用してください。

既存エージェントへの組み込みパターン

パターン向いている状況組み込み方注意点
A. 明示呼び出し図の作業が少ないプロンプトでスキル名を指定呼び忘れを防げない
B. スキル組み込み設計書・文書作成エージェント図変更時の必須工程として設定既存エージェントのコンテキストを使う
C. 専門委譲大きい図、複数候補の比較目的と不変条件を渡して委譲最終採否は親エージェントと人間が行う
D. 品質ゲート公開設計資料を継続的に作るPNG、評価YAML、不変条件検査を完了条件化24/27を自動承認に使わない
責務分担: 既存エージェントは目的・対象読者・論理的不変条件・最終採否を担当する。専門エージェントは図ソース、レンダリング、視覚評価、反復改善を担当する。

入出力の受け渡し

専門エージェントへは「見た目をよくして」だけでなく、変更してはいけない意味を渡す。

入力

diagram_source: docs/order-flow.mmd
purpose: 正常系と補償処理のレビュー
audience: 開発者と運用担当者
invariants:
  - 在庫確保と不正検知は並列
  - 両方成功した場合だけ決済
  - 決済失敗時は在庫を解放
target_score: 24
max_revisions: 3

出力

best_source: candidate-02.mmd
best_png: candidate-02.png
best_svg: candidate-02.svg
evaluation: candidate-02-evaluation.yaml
score: 25
mandatory_violations: []
invariant_check: passed
stop_reason: target_reached
remaining_risks: []

導入チェック

チェック状態はこのページを閉じるまで保持される。

0 / 5 完了
完了の証拠: 「画像を見た」という記述だけでは不十分。生成画像、寸法、視覚所見、評価YAML、不変条件検査を確認する。

運用上の注意

  • 24/27は人間のレビューへ進む目安であり、自動承認基準ではない。
  • AI評価の点数は揺れるため、根拠と重大違反を優先する。
  • 機密設計を外部サービスへ送る場合は、組織のデータ取扱方針に従う。
  • レンダラーとレイアウトエンジンのバージョンを評価結果へ残す。
  • 配置制御だけで改善しない場合は、図の分割または別形式を比較する。

詳細な手順と設定例の正本は usage-and-integration.md を参照する。
